# 07. Complete Practical Example (Complete Example)

This chapter uses "PurchaseOrder" as an example to demonstrate how a complete entity is split into `.entity.yaml`, `.flow.yaml`, and `.views.yaml` under a real business scenario, as well as the final IR generated by the parser.

---

## Directory Structure

```text
/biz-schema
  /Procurement
    PurchaseOrder.entity.yaml   # Data structure definitions
    PurchaseOrder.flow.yaml     # Business logic and workflows
    PurchaseOrder.views.yaml    # Presentation layer layout
    Supplier.entity.yaml        # Associated supplier entity (simplified version)
```

---

## PurchaseOrder.entity.yaml

```yaml
name: PurchaseOrder
label: 採購單
description: "Records documents for company material purchases from suppliers, requires a two-stage process for submission and approval, and returning requires reasons."

fields:
  # Sequential Number: Auto-generated by the system, read-only
  poNumber:
    type: string
    sequence: "PO-{YYYY}{MM}-{SEQ:4}"
    eval:
      readonly: true

  # Status: Driven by the workflow
  status:
    type: enum
    options:
      - value: Draft
        label: 草稿
      - value: Pending
        label: 待審核
      - value: Approved
        label: 已核准
      - value: Rejected
        label: 已退回
    default: Draft

  # Amount fields
  totalAmount: decimal(12,2)!
  discountAmount:
    type: decimal(12,2)
    default: 0

  # Computed field: Not written to DB, calculated by the application layer
  netAmount:
    type: decimal(12,2)
    computed: "totalAmount - discountAmount"

  # Rejection reason: Required and displayed only in the rejected state
  rejectReason:
    type: string
    description: "Reason for rejection, filled by the approver, serving as the basis for subsequent improvement by the filler."
    eval:
      required: "status == 'Rejected'"
      hidden: "status != 'Rejected'"

  # Notes
  notes: string

  # Virtual lookup field: Displays supplier phone number, not saved into DB
  supplierPhone:
    type: lookup
    relation: supplier
    field: phoneNumber

relations:
  supplier: Supplier              # belongsTo → Automatically generates supplierId FK
  items: [PurchaseOrderItem]      # hasMany

indexes:
  - fields: [status, createdAt]
  - fields: [supplierId, status]
```

## PurchaseOrder.flow.yaml

```yaml
name: PurchaseOrder

# Cross-field validations at the entity level (Triggered on every save)
validations:
  - rule: "totalAmount > 0"
    message: "採購金額必須大於零"
  - rule: "discountAmount <= totalAmount * 0.5"
    message: "折扣金額不可超過總額的 50%"

# State machine definitions
workflow:
  statusField: status
  initial: Draft
  terminal: [Approved, Cancelled]   # These two states cannot undergo further transitions

  transitions:
    - action: Submit
      label: 送出審核
      from: [Draft, Rejected]
      to: Pending
      guard:
        validations:
          - rule: "count(items) > 0"
            message: "必須至少有一筆明細才能送審"

    - action: Approve
      label: 核准
      from: [Pending]
      to: Approved

    - action: Reject
      label: 退回
      from: [Pending]
      to: Rejected
      guard:
        requireParams: [rejectReason]

    - action: Cancel
      label: 作廢
      from: [Draft]
      to: Cancelled

# Event Hooks
hooks:
  # Notify external system after approval
  - event: "after:transition:Approve"
    type: webhook
    url: "https://your-platform/webhook/po-approved"
    method: POST
    async: true
    headers:
      X-Api-Key: "secret-token"
    payload:
      poId: "${id}"
      supplierId: "${supplierId}"
      amount: "${totalAmount}"
    retry:
      attempts: 3
      backoff: exponential

  # Record audit logs after any state transition
  - event: "after:transition:*"
    type: webhook
    url: "https://your-platform/webhook/audit-log"
    async: true
    payload:
      entityName: "PurchaseOrder"
      recordId: "${id}"
      status: "${status}"
```

## PurchaseOrder.views.yaml

```yaml
name: PurchaseOrder

views:
  # List View
  list:
    columns: [poNumber, supplier, totalAmount, status, createdAt]
    defaultSort: "-createdAt"
    filterBy: [status, supplier, poNumber]

  # Detail/Form View
  detail:
    actions:
      placement: top

    layout:
      # Card Group: Basic Information
      - type: group
        label: 基本資料
        fields: [poNumber, supplier, supplierPhone, status]

      # Card Group: Amount
      - type: group
        label: 金額資訊
        fields: [totalAmount, discountAmount, netAmount]

      # Pagination Tabs: Line Items and Rejection Details
      - type: tabs
        children:
          - label: 採購明細
            fields: [items]           # Automatically renders Master-Detail sub-table
          - label: 其他資訊
            fields: [notes, rejectReason]
```
